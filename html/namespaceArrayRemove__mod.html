Benchmark moral â›“
  1. The procedures under the generic interface remove take both scalar and vector Pattern arguments. As evidenced by the above benchmark, when the input Pattern is vector of length 1, it is much faster, up to 4X, to pass Pattern as a scalar instead of a whole array of length 1. Note that this benchmark is not relevant to removing substrings from Fortran strings.

Visualization of the benchmark output â›“

Benchmark moral â›“
  1. The procedures under the generic interface genRemoved are functions while the procedures under the generic interface remove are subroutines. From the benchmark results, it appears that the functional interface performs slightly less efficiently than the subroutine interface, despite the two algorithms having the same implementation.
  2. Note that this benchmark does not even include the cost of repeated reallcations, that is, the allocation of Removed happen only once in all tests.
  3. Note that this benchmark considers the worst-case scenario where all elements of the input Array match the input Pattern and must be therefore, removed.

The following is example error message from the ifort compiler,

ArrayRemove_mod@Routines_smod@remove_D1.inc.f90(196): error #6405: The same named entity from different modules and/or program units cannot be referenced. [UNIQUE]
if (present(unique)) unique_def = unique
----------------------------^

Searching this error on the web points to the possibility that the internal representation of entities by ifort has a naming conflict. For now, the remedy was to isolate the use of one of the modules to exactly where it is needed, like,

block
use ArrayUnique_mod, only: genUnique
InstanceNew = genUnique(InstanceNew(1:lenInstanceNew))
end block
