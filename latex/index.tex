Para\+Monte is a serial / parallel library of Monte Carlo routines for sampling mathematical objective functions of arbitrary-\/dimensions, in particular, the posterior distributions of Bayesian models in data science, Machine Learning, and scientific inference, with the design goal of unifying the


\begin{DoxyItemize}
\item {\bfseries{automation}} of Monte Carlo simulations,
\item {\bfseries{user-\/friendliness}} of the library,
\item {\bfseries{accessibility}} from multiple programming environments,
\item {\bfseries{high-\/performance}} at runtime, and,
\item {\bfseries{scalability}} across many parallel processors.

\label{index_ParaMonteNamingConventions}%
\Hypertarget{index_ParaMonteNamingConventions}%
 \doxysection*{\href{\#ParaMonteNamingConventions}{\texttt{ {\bfseries{ Para\+Monte Naming conventions  }} }} }


\begin{DoxyItemize}
\item The {\bfseries{Camel\+Case}} naming style is used throughout the entire Para\+Monte kernel library.
\item {\bfseries{Variable naming conventions}}.
\begin{DoxyItemize}
\item Although the Fortran language is case-\/insensitive, by convention, all scalar variable names begin with a lower case, whereas all vectors, arrays, types, and module names commonly begin with an upper-\/case letter. This rule is not strictly imposed on the public interfaces of the library.
\item The names of variables that represent vectors of values can be suffixed with {\ttfamily Vec} or {\ttfamily Vector} (for example, {\ttfamily Proposal\+Start\+Std\+Vec}, ...) although, this is not recommended since it can conflict with generic programming. Use this naming convention only if the variable is to always represent a vector.
\item The names of variables that represent matrices of values can be suffixed with {\ttfamily Mat} or {\ttfamily Matrix} (for example\+: {\ttfamily Proposal\+Start\+Cor\+Mat}, ...) although, this is not recommended since it can conflict with generic programming. Use this naming convention only if the variable is to always represent a matrix.
\item The names of variables that represent lists of varying-\/size values can be suffixed with {\ttfamily List} (for example, {\ttfamily domain\+Axes\+Name\+List}, ...) although, this is not recommended since such naming can interfere with generic programming.
\item Significant attempt has been made to end all {\ttfamily logical} (boolean) variables with a passive verb, such that the full variable name virtually forms a proposition, that is, an English-\/language statement that should be either {\ttfamily .true.} or {\ttfamily .false.}, for example, {\ttfamily parallelism\+Mpi\+Finalize\+Enabled}.

Occasionally, names that begin with the verb {\ttfamily is} can be also used to label {\ttfamily logical} objects. But as a general rule, names that begin with a verb should be reserved for procedures.
\end{DoxyItemize}
\item {\bfseries{Procedure naming conventions}}.
\begin{DoxyItemize}
\item Procedure (whether function, subroutine, or type-\/bound procedure (class method)) names should be descriptive of the action performed by the procedure. For example,
\begin{DoxyItemize}
\item {\ttfamily gen\+Cov\+Upp} means {\itshape generate (a) covariance upper(-\/triangular matrix)}.
\item {\ttfamily symmetrize\+Mat\+Upp} means {\itshape symmetrize (the input) matrix upper(-\/triangular)}.
\end{DoxyItemize}
\item Procedure names should virtually always begin with a lower-\/case verb.
\item {\bfseries{Function naming conventions}}.
\begin{DoxyItemize}
\item Function names should preferably begin with {\ttfamily gen}. The reasoning is simple; Functions {\bfseries{gen}}erate objects of various types and return them. While it is equally possible to use other verbs to prefix function names, like {\ttfamily get}, such verbs should be reserved for the subroutine equivalents of the same functions. For example, {\ttfamily symmetrize\+Mat\+Upp} is a {\itshape subroutine} that symmetrizes the input upper-\/triangle matrix that has {\ttfamily intent(inout)}. Equivalently, the function {\ttfamily gen\+Mat\+Sym\+From\+Mat\+Upp(\+Mat\+Upp) result(\+Mat\+Sym)} generates a symmetric version of the input matrix and returns it as the function result.
\item Functions that return objects of type {\ttfamily logical} should be preferably prefixed with {\ttfamily is} or be named such that the name begins with a verb and reads as a proposition, evaluating to either {\ttfamily .true.} or {\ttfamily .false.}.
\end{DoxyItemize}
\item {\bfseries{Subroutine naming conventions}}.
\begin{DoxyItemize}
\item The keyword {\bfseries{{\ttfamily gen}}} should be avoided as a prefix for subroutine names since subroutines do not {\bfseries{gen}}erate any objects to return. Subroutines only manipulate and transform arguments passed to them.
\end{DoxyItemize}
\item {\bfseries{Exceptions}} to the above rules are allowed when the procedure name is exceptionally famous or it is very inconvenient to prefix the procedure name with a verb.
\end{DoxyItemize}
\end{DoxyItemize}

 \doxysection*{\href{\#ParaMonteDeveloperWarnings}{\texttt{ {\bfseries{ Developer Guidelines and Warnings  }} }} }


\begin{DoxyItemize}
\item {\bfseries{clarity vs. conciseness}}. ~\newline
 Always prefer clarity to conciseness, specially when choosing variable, argument, or procedure names. ~\newline

\item {\bfseries{Preprocessor MACRO conventions}}. ~\newline

\begin{DoxyItemize}
\item MACRO names, if they are to used as conditions in preprocessing directives, should always read like a proposition that evaluates to either true or false. For example, {\ttfamily CHECK\+\_\+\+ENABLED}, {\ttfamily IS\+\_\+\+WINDOWS}.
\item MACRO names, if they are to used as conditions, must be always given a value of {\ttfamily 1} if true. This is specially important if the macro is defined from within the source code. {\bfseries{Why?}} Such value assignment eliminates reliance of the conditional construct on the {\itshape definition} status of the macro as the condition. For example, the following conditional construct, 
\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{preprocessor}{\#define CHECK\_ENABLED 1}}
\DoxyCodeLine{\textcolor{preprocessor}{}\textcolor{preprocessor}{\#if CHECK\_ENABLED}}
\DoxyCodeLine{\textcolor{preprocessor}{}    \textcolor{fortran_keywordflow}{if} (\textcolor{fortran_operator_logical}{.not.}\textcolor{fortran_function}{ present}(Err) \textcolor{fortran_operator_logical}{.and.} ChoDia(\textcolor{fortran_numeric_literal}{1})\textcolor{fortran_operator_symbols}{ < }\textcolor{fortran_numeric_literal}{0.\_RK})\textcolor{fortran_statement}{ error stop}}
\DoxyCodeLine{\#endif}

\end{DoxyCode}
 executes because {\ttfamily CHECK\+\_\+\+ENABLED} is assigned a value of {\ttfamily 1}.
\end{DoxyItemize}
\item {\bfseries{Preserving the case-\/sensitivity of module/type/procedure/variable names}}. ~\newline
 Although the Fortran programming language is case-\/\+INsensitive, please preserve and ensure the case-\/sensitivity compliance of all object names in all codes and files. This is crucial for the proper hyper-\/linking of different HTML pages of the {\ttfamily Para\+Monte \+:: Kernel} developer library that you are reading right now. For example, if the module name is {\ttfamily \mbox{\hyperlink{namespaceArray__mod}{Array\+\_\+mod}}}, do not write it in any other way like {\ttfamily \mbox{\hyperlink{namespaceArray__mod}{Array\+\_\+mod}}} or {\ttfamily ARRAY\+\_\+\+MOD}. ~\newline

\item {\bfseries{Controlling the runtime checks}}. ~\newline
 The Para\+Monte library has two different build modes (for any build type\+: debug, testing, release). The build mode is determined (by the choice of user) via the preprocessor flag {\ttfamily CHECK\+\_\+\+ENABLED}. If this macro is defined, then certain checks on procedure arguments and validity tests in various modules and procedures of the library are automatically performed at runtime to ensure the validity and accuracy of the calculations. This will however, lower the runtime performance of the library. As such, the Para\+Monte build scripts define the preprocessor macro {\ttfamily CHECK\+\_\+\+ENABLED} by default for the {\ttfamily debug} and {\ttfamily testing} build types, and undefine the macro for the {\ttfamily release} build. For any build type, this behavior can be overriden by the user/developer when building the library. Refer to the help file of the build scripts for more details and instructions. \begin{DoxyWarning}{Warning}
Do {\bfseries{NOT}} lower the case of any uppercase {\ttfamily PURE} procedure attribute. The uppercase {\ttfamily PURE} is a macro that expands to {\ttfamily pure} when runtime checks are disabled, that is, when {\ttfamily CHECK\+\_\+\+ENABLED} is undefined or set to {\ttfamily 0}, otherwise, it is empty. The worst consequence of failing to pay attention to this matter is that the library will not compile and the developer will have to fix any mistakes they have made with {\ttfamily PURE} preprocessor macros. ~\newline
 
\end{DoxyWarning}

\end{DoxyItemize}
