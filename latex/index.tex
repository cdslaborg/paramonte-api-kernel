Para\+Monte is a serial / parallel library of Monte Carlo routines for sampling mathematical objective functions of arbitrary-\/dimensions, in particular, the posterior distributions of Bayesian models in data science, Machine Learning, and scientific inference, with the design goal of unifying the


\begin{DoxyItemize}
\item {\bfseries{automation}} of Monte Carlo simulations,
\item {\bfseries{user-\/friendliness}} of the library,
\item {\bfseries{accessibility}} from multiple programming environments,
\item {\bfseries{high-\/performance}} at runtime, and,
\item {\bfseries{scalability}} across many parallel processors.

\begin{DoxyItemize}
\item The {\bfseries{Camel\+Case}} naming style is used throughout the entire Para\+Monte kernel library.
\item {\bfseries{Variable naming conventions}}.
\begin{DoxyItemize}
\item Although the Fortran language is case-\/insensitive, by convention, all scalar variable names begin with a lower case, whereas all vectors, arrays, types, and module names commonly begin with an upper-\/case letter. This rule is not strictly imposed on the public interfaces of the library.
\item All constants and parameters should be generally typed upper-\/case entirely. ~\newline

\item The names of variables that represent vectors of values can be suffixed with {\ttfamily Vec} or {\ttfamily Vector} (for example, {\ttfamily Proposal\+Start\+Std\+Vec}, ...) although, this is not recommended since it can conflict with generic programming. Use this naming convention only if the variable is to always represent a vector.
\item The names of variables that represent matrices of values can be suffixed with {\ttfamily Mat} or {\ttfamily Matrix} (for example\+: {\ttfamily Proposal\+Start\+Cor\+Mat}, ...) although, this is not recommended since it can conflict with generic programming. Use this naming convention only if the variable is to always represent a matrix.
\item The names of variables that represent lists of varying-\/size values can be suffixed with {\ttfamily List} (for example, {\ttfamily domain\+Axes\+Name\+List}, ...) although, this is not recommended since such naming can interfere with generic programming.
\item Significant attempt has been made to end all {\ttfamily logical} (boolean) variables with a passive verb, such that the full variable name virtually forms a proposition, that is, an English-\/language statement that should be either {\ttfamily .true.} or {\ttfamily .false.}, for example, {\ttfamily parallelism\+Mpi\+Finalize\+Enabled}.

Occasionally, names that begin with the verb {\ttfamily is} can be also used to label {\ttfamily logical} objects. But as a general rule, names that begin with a verb should be reserved for procedures.
\end{DoxyItemize}
\item {\bfseries{Procedure naming conventions}}.
\begin{DoxyItemize}
\item Procedure (whether function, subroutine, or type-\/bound procedure (class method)) names should be descriptive of the action performed by the procedure. For example,
\begin{DoxyItemize}
\item {\ttfamily gen\+Cov\+Upp} means {\itshape generate (a) covariance upper(-\/triangular matrix)}.
\item {\ttfamily symmetrize\+Mat\+Upp} means {\itshape symmetrize (the input) matrix upper(-\/triangular)}.
\end{DoxyItemize}
\item Procedure names should virtually always begin with a lower-\/case verb.
\item {\bfseries{Function naming conventions}}.
\begin{DoxyItemize}
\item Function names should preferably begin with {\ttfamily gen}. The reasoning is simple; Functions {\bfseries{gen}}erate objects of various types and return them. While it is equally possible to use other verbs to prefix function names, like {\ttfamily get}, such verbs should be reserved for the subroutine equivalents of the same functions. For example, {\ttfamily symmetrize\+Mat\+Upp} is a {\itshape subroutine} that symmetrizes the input upper-\/triangle matrix that has {\ttfamily intent(inout)}. Equivalently, the function {\ttfamily gen\+Mat\+Sym\+From\+Mat\+Upp(\+Mat\+Upp) result(\+Mat\+Sym)} generates a symmetric version of the input matrix and returns it as the function result. ~\newline
 {\bfseries{Exceptions}} to this naming convention are allowed, for example, when a procedure is expected to exist only as a function (and not a subroutine). An example of such an exception is \mbox{\hyperlink{interfaceArrayReplace__mod_1_1replace}{replace}}.
\item Functions that return objects of type {\ttfamily logical} should be preferably prefixed with {\ttfamily is} or be named such that the name begins with a verb and reads as a proposition, evaluating to either {\ttfamily .true.} or {\ttfamily .false.}.
\end{DoxyItemize}
\item {\bfseries{Subroutine naming conventions}}.
\begin{DoxyItemize}
\item The keyword {\bfseries{{\ttfamily gen}}} should be avoided as a prefix for subroutine names since subroutines do not {\bfseries{gen}}erate any objects to return. Subroutines only manipulate and transform arguments passed to them.
\end{DoxyItemize}
\item {\bfseries{Exceptions}} to the above rules are allowed when the procedure name is exceptionally famous or it is very inconvenient to prefix the procedure name with a verb.
\end{DoxyItemize}

\begin{DoxyItemize}
\item {\bfseries{Case-\/sensitivity}}.
\begin{DoxyItemize}
\item Although the Fortran programming language is case-\/\+INsensitive, please preserve and ensure the case-\/sensitivity compliance of all object names including module, type, procedure, and variable names in all codes and files. This is crucial for the proper hyper-\/linking of different HTML pages of the {\ttfamily Para\+Monte \+:: Kernel} developer library that you are reading right now, that is generated by Doxygen. For example, if the module name is {\ttfamily \mbox{\hyperlink{namespaceMath__mod}{Math\+\_\+mod}}}, do not write it in any other way like {\ttfamily math\+\_\+mod} or {\ttfamily MATH\+\_\+\+MOD}.
\item If a variable or procedure argument in a generic interface can be both scalar and array, it should be preferably capitalized since a scalar can be envisioned as particular case of an array with one element.
\end{DoxyItemize}
\item {\bfseries{Clarity vs. conciseness}}.
\begin{DoxyItemize}
\item Always prefer clarity to conciseness, specially when choosing variable, argument, or procedure names.
\end{DoxyItemize}
\item {\bfseries{Preprocessor MACRO conventions}}.
\begin{DoxyItemize}
\item MACRO names, if they are to used as conditions in preprocessing directives, should always read like a proposition that evaluates to either true or false. For example, {\ttfamily CHECK\+\_\+\+ENABLED}, {\ttfamily IS\+\_\+\+WINDOWS}.
\item MACRO names, if they are to used as conditions, must be always given a value of {\ttfamily 1} if true. This is specially important if the macro is defined from within the source code. {\bfseries{Why?}} Such value assignment eliminates reliance of the conditional construct on the {\itshape definition} status of the macro as the condition. For example, the following conditional construct executes because {\ttfamily CHECK\+\_\+\+ENABLED} is assigned a value of {\ttfamily 1}. ~\newline

\item {\bfseries{Controlling the runtime checks}}.
\begin{DoxyItemize}
\item The Para\+Monte library has two different build modes (for any build type\+: {\ttfamily debug}, {\ttfamily testing}, {\ttfamily release}). The build mode is determined (by the choice of user) via the preprocessor flag {\ttfamily CHECK\+\_\+\+ENABLED}. If this macro is defined, then certain checks on procedure arguments and validity tests in various modules and procedures of the library are automatically performed at runtime to ensure the validity and accuracy of the calculations. This will however, lower the runtime performance of the library. As such, the Para\+Monte build scripts define the preprocessor macro {\ttfamily CHECK\+\_\+\+ENABLED} by default for the {\ttfamily debug} and {\ttfamily testing} build types, and undefine the macro for the {\ttfamily release} build. For any build type, this behavior can be overridden by the user/developer when building the library. Refer to the help file of the build scripts for more details and instructions. \begin{DoxyWarning}{Warning}
Do {\bfseries{NOT}} lower the case of any uppercase {\ttfamily PURE} procedure attribute. The uppercase {\ttfamily PURE} is a macro that expands to {\ttfamily pure} when runtime checks are disabled, that is, when {\ttfamily CHECK\+\_\+\+ENABLED} is undefined or set to {\ttfamily 0}, otherwise, it is empty. The worst consequence of failing to pay attention to this matter is that the library will not compile and the developer will have to fix any mistakes they have made with {\ttfamily PURE} preprocessor macros.
\end{DoxyWarning}

\item When the library is built with tests enabled, a number of fatal {\ttfamily error stop} statements are disabled in some procedures (like the Para\+Monte samplers), in particular, when the library is built in parallel mode. Such temporary disabling of global program halts is needed to ensure the error signals are propagated correctly with the library and across multiple processors. However, such extra testing communications (in particular, in parallel mode) can be very costly and therefore, {\bfseries{the library should never be built for production release with any tests of the library enabled}}.
\end{DoxyItemize}
\item {\bfseries{Data hiding and privacy}}
\begin{DoxyItemize}
\item Module entities should be all preferably {\ttfamily public} unless an entity is guaranteed to never be needed outside the module, in which case, it should be given the {\ttfamily private} attribute. An example of such guaranteed privacy is a type-\/bound procedure. The reason for keeping all module entities {\ttfamily public} is simple. Generic interfaces cannot be passed as dummy arguments. Therefore, all module procedures, even though accessible via generic interfaces, should be kept public.
\item Avoid declaring and using global module variables, unless the variable is guaranteed to remain unchanged after program initialization. If a module variable is only initialized once throughout the entire life of the program, then it should be preferably prefixed with {\ttfamily mc\+\_\+} to indicate the variable is a {\bfseries{m}}odule runtime {\bfseries{c}}onstant (not a {\ttfamily parameter}, since parameters are compile-\/time constants). in which case, it should be given the {\ttfamily private} attribute. An example of such guaranteed privacy is a type-\/bound procedure. The reason for keeping all module entities {\ttfamily public} is simple. Generic interfaces cannot be passed as dummy arguments. Therefore, all module procedures, even though accessible via generic interfaces, should be kept public.
\end{DoxyItemize}

\item {\bfseries{Doxygen custom command orderings}}.
\begin{DoxyItemize}
\item The Doxygen tag {\ttfamily \textbackslash{}brief} must always be the first line of the documentation of modules, types, and procedures. Example\+: \mbox{\hyperlink{namespaceMath__mod}{Math\+\_\+mod}}.
\item The Doxygen tag {\ttfamily \textbackslash{}details}, if it exists, must always immediately follow the Doxygen tag {\ttfamily \textbackslash{}brief}. Example\+: \mbox{\hyperlink{namespaceMath__mod}{Math\+\_\+mod}}.
\item The Doxygen tag {\ttfamily \textbackslash{}param}, if any number of it exists, must always immediately follow the Doxygen tag {\ttfamily \textbackslash{}brief} (or {\ttfamily \textbackslash{}details} if it exists). Example\+: gen\+Mean().
\item The Doxygen tag {\ttfamily \textbackslash{}return}, must be exclusively used to indicate the return value of functions. If it exists, it must appear immediately after the set of {\ttfamily \textbackslash{}param} tags. Example\+: gen\+Mean().
\item If a generic interface is being documented, the Para\+Monte custom command {\ttfamily \textbackslash{}interface} must appear immediately after the Doxygen {\ttfamily \textbackslash{}return}, {\ttfamily \textbackslash{}param}, {\ttfamily \textbackslash{}details}, or {\ttfamily \textbackslash{}brief} tags in the specified order, if any exists.
\item The Doxygen tag {\ttfamily \textbackslash{}warning}, if any number of it exists, must immediately follow the Doxygen tag {\ttfamily \textbackslash{}return} if it exists, otherwise {\ttfamily \textbackslash{}param} if it exists, otherwise {\ttfamily \textbackslash{}details} if it exists, otherwise {\ttfamily \textbackslash{}brief}. The {\ttfamily \textbackslash{}warning} tag must be used to highlight situations that require special attention of the user, otherwise, there is a danger for the code section being documented to not behave normally as one may expect.
\item The Doxygen tag {\ttfamily \textbackslash{}attention} has the same functionality and usage as {\ttfamily \textbackslash{}warning}. Therefore, {\ttfamily \textbackslash{}warning} should be preferred wherever {\ttfamily \textbackslash{}attention} is needed. Exceptions are allowed and if they occur, the same documentation conventions as those of {\ttfamily \textbackslash{}warning} also apply to the tag {\ttfamily \textbackslash{}attention}.
\item The Doxygen tag {\ttfamily \textbackslash{}remark}, if any number of it exists, must immediately follow the Doxygen tag {\ttfamily \textbackslash{}warning} if it exists, otherwise the Doxygen tag {\ttfamily \textbackslash{}return} if it exists, otherwise {\ttfamily \textbackslash{}param} if it exists, otherwise {\ttfamily \textbackslash{}details} if it exists, otherwise {\ttfamily \textbackslash{}brief}. The tag {\ttfamily \textbackslash{}remark} should be reserved for explaining behavior that is {\bfseries{directly}} related to the code segment being documented, but its knowledge is not so critical as warrant the use of a {\ttfamily \textbackslash{}warning} tag.
\item The Doxygen tag {\ttfamily \textbackslash{}note}, if it exists, must appear after all {\ttfamily \textbackslash{}warning} and {\ttfamily \textbackslash{}attention} and {\ttfamily \textbackslash{}remark} tags and immediately before the Para\+Monte custom command tag {\ttfamily \textbackslash{}see} if it exists, otherwise immediately before {\ttfamily \textbackslash{}example} for examples (if it exists).
\item The Doxygen tag {\ttfamily \textbackslash{}see}, if it exists, must appear after all {\ttfamily \textbackslash{}warning} and {\ttfamily \textbackslash{}remark} and {\ttfamily \textbackslash{}note} tags.
